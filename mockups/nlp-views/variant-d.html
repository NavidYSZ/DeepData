Variante D — Parent-Pillar Hypothese

Statt einer neuen Topical-Map-Sicht: fixt die Hub/Child-Logik direkt im Page Profile. Wenn das Tool eine Seite als Child-Page erkennt, projiziert die LLM zusätzlich: welche Pillar-Page müsste über dieser hängen, und welche Geschwister-Pages müssten daneben existieren? Beantwortet "wo sitzt diese Seite im implizierten Cluster?" — was der aktuelle Three-Pillar-Salat (Zahnarzt + Tornesch + Brand) nicht beantwortet.

Interpretation

Die Seite ist als Brand/Landing klassifiziert (Child-Page), behandelt aber gleichzeitig 9 Subtopics mit jeweils shallow Coverage. Daraus folgen zwei mögliche Lesarten:

  1. Sie wäre eigentlich ein Pillar, aber zu dünn ausgeführt — dann müsste sie tiefer werden (Subtopics ausarbeiten) und die Service-Pages als Children dazubekommen.
  2. Sie bleibt Brand/Landing, dann fehlt der echte Pillar "Zahnarzt in Tornesch" als Übersichtsseite — und die 9 Subtopics gehören als eigene Cluster-Pages aufgemacht.

Beide Lesarten landen praktisch beim gleichen To-Do: fehlende Service-Pages anlegen. Genau das macht der aktuelle Entity-Graph nicht sichtbar, weil er die drei gleichberechtigten Pillars (Zahnarzt + Tornesch + Sabah Dentalmedizin) zeigt statt der Hierarchie-Frage.

Was diese Variante kann (und was nicht)

✓ Behebt das konkrete Problem aus deinem Beispiel: das 3-Pillar-Dreieck verwirrt, weil keine Hierarchie sichtbar wird. Diese Variante stellt die Hierarchie explizit her.

✓ Kleinster Eingriff von den drei "Empfehlungs-Optionen": passt komplett ins Page Profile, kein neues View-Konzept.

✓ Funktioniert auch ohne separate Tree-Ansicht — also gut kombinierbar mit "Entity Map bleibt das Hauptbild".

✗ Zeigt keine URL-Slugs, H1s oder konkreten Page-Empfehlungen pro Subtopic (das macht A+B+C).

✗ Bleibt ein 2-Ebenen-Blick (Parent + Siblings), keine vollständige Sitemap.

! Risiko: Hypothesen werden oft als "Fakten" gelesen — UI muss klar machen, dass der Parent vorgeschlagen ist und vom LLM stammt, nicht gecrawlt wurde.
